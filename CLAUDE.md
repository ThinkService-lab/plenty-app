## What this app is

PlentyMeals is a single-page AI meal suggestion app. Users enter ingredients they have at home and receive 3 personalised meal suggestions generated by Claude Haiku. The tone is empowering and abundant ‚Äî the word "only" is banned from the AI prompt.

---

## Architecture ‚Äî one file rule

**The entire frontend lives in `index.html`.** There is no build step, no framework, no bundler. Every CSS rule, HTML element, and JS function is in this one file. Do not split it.

The backend is five Vercel serverless functions in `/api/`:
- `meals.js` ‚Äî proxies the Anthropic API call; checks usage limits for authenticated users
- `scan.js` ‚Äî accepts a base64 image, calls Claude Haiku vision, returns identified ingredients as JSON
- `usage.js` ‚Äî exports `checkAndIncrementUsage(accessToken)` (imported by `meals.js`); also serves GET `/api/usage` for the frontend to fetch current usage count
- `subscribe.js` ‚Äî saves emails to Google Sheets
- `photo.js` ‚Äî unused, leave it

## Critical JS rules

**All onclick functions must be on `window.*`** ‚Äî functions are defined inside a `<script>` block, not globally. Any new function called via `onclick="..."` in HTML must be assigned as `window.functionName = function() {}` or it will throw `undefined is not a function`.

**Do not use `innerHTML =` on user-controlled input** ‚Äî the security hook will block your edit. Use the template literal pattern already established for `card` markup (set via `.innerHTML` on createElement'd divs with server-returned data, which is acceptable here).

**JS template string edits** ‚Äî the meal card scaffold is built inside `renderResults()` as template literals. Only touch the HTML scaffold strings (what elements exist), never the data logic, API calls, event handlers, or function signatures.

## Profile strip inputs

Four selects in `#formSection > .profile-strip`, read in `generateMeals()`:

| ID | Purpose | Default |
|----|---------|---------|
| `cuisinePref` | Cuisine type (13 options, "any" = no constraint) | `any` |
| `dietPref` | Dietary restriction | `none` |
| `skillLevel` | Cooking skill | `beginner` |
| `servings` | Number of people | `4` |

All four are injected into the AI prompt as: `Cuisine: X | Diet: X | Skill: X | Servings: X`

**Cuisine adaptation rule** ‚Äî the prompt instructs the AI: if the chosen cuisine doesn't naturally fit the ingredients, suggest the closest honest adaptation and name it clearly (e.g. "African-inspired pasta bake", not "Nigerian jollof"). This prevents misleading meal names when ingredients and cuisine don't match.

## Key behaviours

- **Form hides on results** ‚Äî `#formSection` is hidden when `resultsSection` becomes active. `resetApp()` restores it.
- **Loading state** ‚Äî aurora blobs + dual rings + shimmer skeleton cards. Class `.loading-state.active` shows it.
- **Slow load fallback** ‚Äî after 15s, `.loading-text` text is replaced with a reassurance message.
- **Rate meals influence next generation** ‚Äî liked/disliked meals are tracked in `ratingContext` and injected into the next prompt.
- **Health conditions** ‚Äî stored in `selectedConditions` (Set). Rules are injected into the AI prompt via `buildHealthRules()`.

## Anonymous user gate

Anonymous users get 2 lifetime tries (stored in `localStorage` key `plenty_anon_count`). **No time-based reset** ‚Äî permanent until they sign in. This is intentional to push signups.

When `getAnonCount() >= 2` and user tries to generate:
1. `window.showSignInGate()` is called ‚Äî hides `#generateBtn`, shows `#signinGate` (a styled card with title, explanation, and "Sign in to continue" button)
2. Login modal also opens on top
3. If user dismisses the modal, the gate card remains on the page
4. On sign-in: `renderAuthChip()` calls `hideSignInGate()` ‚Äî gate hidden, generate button restored
5. On sign-out: `signOut()` calls `resetAnonCount()` + `hideSignInGate()` ‚Äî fresh 2 tries

`#usageIndicator` shows anon count for logged-out users ("X of 2 free tries left ‚Äî sign in for more") and Firestore count for logged-in free users. Updated after every generation.

## Ingredient photo scan

Users tap the üì∑ button in the ingredient input row to scan a photo of their fridge or countertop.

**Flow:** camera button ‚Üí scan options (Take photo / Upload from gallery) ‚Üí hidden `<input type="file">` ‚Üí `handleScanFile()` ‚Üí resize via Canvas API ‚Üí POST to `/api/scan` ‚Üí confirmation panel with pre-ticked checkboxes ‚Üí `addScannedIngredients()` ‚Üí merged into tag list.

**Key scan JS functions (all on `window.*`):**
- `window.toggleScanOptions()` ‚Äî shows/hides the two scan option buttons; blocked while `isScanning === true`
- `window.handleScanFile(input)` ‚Äî orchestrates resize ‚Üí API call ‚Üí confirmation panel render
- `window.addScannedIngredients()` ‚Äî reads checked items, lowercases names, calls `window.quickAdd(name, qty)` for each
- `window.dismissScan()` ‚Äî hides all scan UI, resets file input values

**`resizeImage(file, maxPx)`** ‚Äî internal (not on window). Uses Canvas API to resize to max 800px on longest dimension. Always encodes as JPEG via `canvas.toDataURL('image/jpeg')`. Strips the data-URI prefix before returning: `dataUri.split(',')[1]` ‚Äî the Claude API needs raw base64, not a data-URI string.

**`isScanning` flag** ‚Äî set `true` at the start of `handleScanFile`, `false` on every exit path (success, error, null result). Prevents toggling the scan options while a scan is in progress.

**`api/scan.js`** ‚Äî mirrors `api/meals.js` for rate limiting and CORS. Accepts `{ image: "<base64>", mimeType: "image/jpeg" }`. The Claude API `content` field is an **array** (`[imageBlock, textBlock]`), unlike `meals.js` which passes a string. Returns `{ ingredients: [{name, quantity?}, ...] }`.

**Duplicate prevention** ‚Äî `window.quickAdd()` does a **case-sensitive** `===` check on ingredient names. `addScannedIngredients()` lowercases all names from Claude before calling `quickAdd()` ‚Äî this is intentional and must not be removed.

## What NOT to change without care

### `sanitizeString` allowlist in `api/meals.js`
The allowlist regex includes `{`, `}`, `[`, `]`, `"`, `'`, `!`, `?`, `#`, `@` deliberately. These are needed so that the JSON schema instructions in the AI prompt survive sanitisation intact. If these characters are stripped, Claude receives a garbled schema and returns `ingredients_used` and `steps` as plain strings instead of arrays ‚Äî causing `.map is not a function` errors in `renderResults`. The HTML-strip step before the allowlist already handles XSS.

### `charCodeAt()` control character cleaning in `api/meals.js`
The AI response is JSON. Claude occasionally returns invisible control characters (`\x00`‚Äì`\x1F`) inside strings that break `JSON.parse()`. The normal fix is a regex like `/[\x00-\x1F]/g` ‚Äî but that regex, written with literal characters in the file, was **silently corrupting Vercel's compilation** with no error message. The workaround is a character-by-character loop checking `charCodeAt(i) < 32`. It looks verbose but it is a real Vercel edge case. Do not replace it with a regex.

### `claude-haiku-4-5-20251001` instead of a more capable model
Vercel's free tier has a **10-second function timeout**. Claude Sonnet consistently hits that limit generating the full meal JSON response. Haiku is fast enough to complete within the timeout. This is not a cost decision ‚Äî it is a deployment constraint. Do not upgrade the model without first upgrading the Vercel plan or verifying the timeout.

### `max_tokens: 1800` passed from the frontend
This is higher than expected for a meal card. When a user selects health conditions (e.g. Diabetes + Heart Disease), the AI generates additional `health_insights` per meal on top of the base response. At the original limit of 1200, responses were truncating mid-JSON and causing parse failures in production. 1800 gives enough headroom for the worst-case prompt. It is passed from the frontend so `meals.js` stays flexible without hardcoding it.

### Feedback POST goes directly to Google Apps Script URL from the frontend
There is no Vercel function for feedback. The frontend calls the Google Apps Script URL directly with query params. This is intentional to avoid an extra serverless function and cold start latency.

## Firebase auth + usage tracking

**Status:** Shipped to production on `main`.

**Feature summary:**
- Anonymous users get 2 lifetime tries (tracked in `localStorage` key `plenty_anon_count`) ‚Äî no time reset, permanent gate
- Logged-in free users get 2 generations/day ‚Äî enforced server-side via Firestore atomic transaction
- Google OAuth + email magic link login (no password)
- Paywall modal on limit hit (`data.error === 'limit_reached'` from `/api/meals`)
- Currency auto-detected from browser locale (`en-CA` ‚Üí CAD, `en-GB` ‚Üí GBP, etc.)

**Apple Sign In ‚Äî not yet added, required for App Store:**
Apple mandates Sign in with Apple if any third-party OAuth (e.g. Google) is offered. Firebase Auth supports it. Must be added before iOS App Store submission.

**Firebase env vars (required in `.env.local` and Vercel dashboard):**
- `FIREBASE_PROJECT_ID` ‚Äî server-side only
- `FIREBASE_CLIENT_EMAIL` ‚Äî server-side only (service account)
- `FIREBASE_PRIVATE_KEY` ‚Äî **secret, server-side only, never in `index.html`**
- Firebase web config (`apiKey`, `authDomain`, `projectId`) ‚Äî hardcoded in `index.html` (public, safe to expose)

**Critical env var formatting (Vercel CLI):** Always use `printf '%s'` not `echo` to avoid a trailing newline. For `FIREBASE_PRIVATE_KEY`, convert `\n` literals to real newlines before adding:
```bash
cat key.txt | python3 -c "import sys; print(sys.stdin.read().replace('\\\\n', '\\n'), end='')" | vercel env add FIREBASE_PRIVATE_KEY production
```
A malformed private key causes Firestore calls to fail with `16 UNAUTHENTICATED` (even though `verifyIdToken` still works, because it uses Firebase's public key, not the service account key).

**`authDomain` must be the Firebase Hosting domain** ‚Äî `plenty-meals.firebaseapp.com`. Email magic links go to `{authDomain}/__/auth/action?...`; only Firebase Hosting serves that route. Setting it to the Vercel domain causes magic links to silently fail.

**Anonymous counter helpers must be in the non-module script** ‚Äî `getAnonCount`, `incrementAnonCount`, `resetAnonCount` are in the plain `<script>` block, not in `<script type="module">`. Firebase CDN ES module imports are deferred; if a user clicks "Generate" before the CDN resolves, module-scoped functions are `undefined`. The anon counter only needs `localStorage`, so it has no Firebase dependency and belongs in the synchronous script.

**npm dependency:** `firebase-admin` in `package.json` (only new dependency added for auth).

**Firebase JS on frontend:** ES module imports from Google CDN at pinned version:
```
https://www.gstatic.com/firebasejs/10.12.2/firebase-app.js
https://www.gstatic.com/firebasejs/10.12.2/firebase-auth.js
```
Loaded via `<script type="module">` ‚Äî all functions assigned to `window.*` for onclick access.

**Server-side:** `firebase-admin` npm package handles token verification (`verifyIdToken`) and Firestore transactions (`usage` collection, doc pattern: `{uid}_{YYYY-MM-DD}`).

**Do not use `node --check` on `index.html`** ‚Äî Node can't parse HTML files. Use `vercel dev` and check the browser console instead. `node --check` is valid for `.js` files only.

## Local dev

```bash
vercel dev          # starts at http://localhost:3000
```

Requires `.env.local` with:
- `ANTHROPIC_API_KEY`
- `FIREBASE_PROJECT_ID`
- `FIREBASE_CLIENT_EMAIL`
- `FIREBASE_PRIVATE_KEY` (with real newlines, not `\n` literals)

Pull the current values from Vercel with `vercel env pull .env.local` (run from the project root). Note: `vercel env pull` **overwrites** `.env.local` ‚Äî if you have local-only values, re-add them after pulling.
